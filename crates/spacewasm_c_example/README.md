# SpaceWasm example C API

A C-linkable build of the [`spacewasm`](../..) WebAssembly 1.0 interpreter.

This directory is a **reference integrator crate**. The reusable FFI logic lives
in [`spacewasm_c_api`](../spacewasm_c_api); this crate simply fixes the
configuration and emits the concrete `spacewasm_*` C symbols.

## Layers

- `spacewasm`: Core `#![no_std]` interpreter (pure Rust).
- `spacewasm_c_api`: FFI support + the `spacewasm_*` C entry points
  - This includes most of the C wrappers expect for the implementation of the allocators
- This crate implements the allocators in Rust and exports the C functions from the FFI layer

The interpreter capacities `MAX_CONTROL_FRAMES` and `MAX_STACK_DEPTH` are chosen
when `spacewasm_c_api` compiles, from an environment variable (see below), because C
cannot supply const-generic parameters. The maximum number of compiled code
pages is instead a runtime argument to `spacewasm_builder_finish`. The two
allocators C cannot supply are provided by the integrator as linker symbols;
everything else is exposed through the C functions.

### Changing the configuration

The maximum number of code pages is passed at runtime as the `max_code_pages`
argument to `spacewasm_builder_finish`, so it needs no rebuild.

The remaining capacities are compile-time constants generated by
[`spacewasm_c_api`'s `build.rs`](../spacewasm_c_api/build.rs) via  configuration file which can be specified with the environment variable `SPACEWASM_CONFIG`, each with a default:

| Constant                        | Default |
| ------------------------------- | ------- |
| `MAX_CONTROL_FRAMES`            | 64      |
| `MAX_STACK_DEPTH`               | 256     |
| `GLOBAL_ALLOCATOR_MAX_PAGES`    | 16      |
| `GLOBAL_ALLOCATOR_PAGE_SIZE`    | 8192    |

To set them for the build, add them to a `.rs` source file as `pub const` values and then include that config file with the `SPACEWASM_CONFIG` environment variable before rebuilding:

```sh
SPACEWASM_CONFIG=my_config.rs cargo build -p spacewasm_c_api
```

The C header and ABI are unchanged (handles are opaque), so C callers do **not**
recompile. The capacities cannot be set from C at runtime; that is why the
integrator layer exists.

### Flight builds

This reference uses a std-backed allocator so it runs on a host for testing. A
flight build should substitute a deterministic allocator and compile with
`panic = "abort"` (a Rust panic crossing the FFI boundary in `no_std` is
undefined behavior).

## Usage from C

See [`examples/ctest.c`](examples/ctest.c)

## Building & testing

```sh
cargo build -p spacewasm_c_example            # produces libspacewasm_c.{a,dylib/so}
cargo test  -p spacewasm_c_example            # Rust-driven ABI tests + C link/run test
```

`tests/c_api.rs` drives the generated `extern "C"` functions directly.
`tests/c_abi.rs` compiles [`examples/ctest.c`](examples/ctest.c) against the
header ([`../spacewasm_c_api/include/spacewasm.h`](../spacewasm_c_api/include/spacewasm.h))
and staticlib and runs it (skipped if no C compiler is present).

### Regenerating the C header

[`include/spacewasm.h`](../spacewasm_c_api/include/spacewasm.h) is generated from
the Rust source by `spacewasm_c_api`'s build script (cbindgen) and committed. To
refresh it after changing the C surface, build `spacewasm_c_api` with the
`codegen` feature:

```sh
cargo build -p spacewasm_c_api --features codegen
```

Ordinary builds do not need the feature (or cbindgen); they consume the
committed header as-is.
